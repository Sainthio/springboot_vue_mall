## 第8页：核心功能 - 防超卖机制

### 问题场景
- 商品库存只有100件
- 1000人同时下单
- 如何保证不会卖出超过100件？

### 解决方案：数据库行锁

```sql
UPDATE shangpinxinxi 
SET alllimittimes = alllimittimes - #{buynumber}
WHERE id = #{goodid} 
  AND alllimittimes >= #{buynumber}
```

### 关键点
- `WHERE alllimittimes >= #{buynumber}`：只有库存够才扣减
- 数据库层面保证原子性
- 使用行锁防止并发冲突

### 流程图
```
检查库存 → 扣减库存（行锁） → 创建订单 → 返回成功
   ↓ 库存不足
 返回失败
```

## 第9页：核心功能 - 限购机制

### 问题场景
- 防止同一用户重复购买
- 每人限购1件

### 解决方案

```java
// 检查用户是否已购买
EntityWrapper<OrdersEntity> wrapper = new EntityWrapper<>();
wrapper.eq("userid", userid);
wrapper.eq("goodid", goodid);
wrapper.eq("status", "已支付");

int count = ordersService.selectCount(wrapper);
if(count > 0) {
    return R.error("您已购买过该商品，不能重复购买");
}
```

### 流程
```
用户下单 → 查询历史订单 → 判断是否已购买
   ↓ 未购买          ↓ 已购买
 允许购买         拒绝购买
```

## 第13页：性能测试

### 测试工具：Apache JMeter

### 并发测试结果

| 并发数 | 平均响应时间 | 成功率 | 错误率 | 评价 |
|--------|-------------|--------|--------|------|
| 100 | 200ms | 100% | 0% | ✅ 优秀 |
| 500 | 350ms | 100% | 0% | ✅ 良好 |
| 1000 | 500ms | 98% | 2% | ✅ 可接受 |
| 5000 | 1200ms | 85% | 15% | ⚠️ 部分超时 |
| 10000 | 2500ms | 70% | 30% | ❌ 大量超时 |

### 测试结论
- ✅ 支持1000并发用户
- ✅ 行锁机制有效防止超卖
- ✅ 无库存超卖现象
